# 差別化アイデアメモ（vs Zenn / Qiita / note）

2026-07-16 時点の構想。軸 = 「ブログとして戦わず、**サービス分析の構造化データベース**に振り切る」。

## 現状すでにある差別化

1. 記事がデータベースのレコード（解剖スコア4軸・techStack confidence3段階+根拠URL・出典+閲覧日）。CIが整合性を強制
2. 事実/推測の分離がUIレベルで見える（fact/guessカラウト・確度バッジ）= 編集ポリシーがデザイン
3. ja/en完全バイリンガル（Zenn/Qiita/noteは実質日本語圏のみ）
4. SEO資産・ブランドが自ドメインに積み上がる

弱点 = 集客力ゼロスタート（プラットフォームの初日流入がない）。

## 強化アイデア（優先順）

1. **技術スタック横断ページ（本命・安い）** — techStack frontmatter から `/[locale]/tech/<slug>`
   （例: 「Vercelを採用しているサービス一覧」）を自動生成。記事が増えるほど
   「日本版BuiltWith+分析記事」の資産になる。ロングテールSEOの入口。既存tagページとほぼ同型で流用可
2. **定点観測（再解剖）** — 同一サービスを半年後に再スコアリングし差分を見せる。
   単発レビューに対する「時系列データを持つメディア」化。lastVerified/updatedAt と相性良
3. **比較解剖フォーマット** — 2サービスの scorecard/techstack を並置する対決記事
   （例: Nani翻訳 vs DeepL）。構造化データがあるから安く作れる
4. **データのオープン化** — 全記事の構造化データを JSON（例: `/api/anatomy.json`）で公開。
   開発者の引用・集計を促し、被リンク獲得装置にする
5. **Zenn/noteは衛星チャネル** — 深掘り技術編をZennに書き本体へリンク（promoスキルの流れに組込）。
   集客弱点をプラットフォームで補う住み分け
